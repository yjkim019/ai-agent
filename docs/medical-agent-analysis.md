## 1. 시스템 개요

본 프로젝트는 FastAPI, LangChain, LangGraph를 기반으로 구축된 의료 AI 에이전트 시스템이다. 사용자가 건강 관련 질문을 입력하면, 에이전트가 질문의 의도를 스스로 분석하여 외부 데이터 소스(Elasticsearch, 식약처 API, 건강보험심사평가원 API)에서 필요한 정보를 검색하고, 이를 종합하여 의료 상담 답변을 생성한다.

핵심적인 설계 원칙은 에이전트가 **어떤 도구를 어떤 순서로 호출할지 스스로 결정**한다는 점이다. 미리 정해진 규칙 기반 흐름이 아니라, LLM이 질문의 맥락을 파악하여 적절한 도구 조합을 자율적으로 선택한다.

---

## 2. 전체 동작 흐름

### 2-1. 요청 수신

사용자가 `/api/v1/chat` 엔드포인트로 `thread_id`와 `message`를 전송한다. `thread_id`는 대화 세션을 식별하는 키로, 이를 통해 이전 대화 맥락을 유지한다.

### 2-2. LLM 및 저장소 초기화

`AgentService`가 GPT-4o 모델을 LLM으로 초기화하고, SQLite 기반 `AsyncSqliteSaver`를 대화 이력 저장소로 연결한다. SQLite를 사용하므로 서버가 재시작되어도 이전 대화 기록이 보존된다.

### 2-3. 에이전트 생성

`create_medical_agent`에서 LLM, 3개 도구, 시스템 프롬프트, 응답 포맷(`ChatResponse`)을 조합하여 LangGraph 에이전트를 생성한다. LLM은 판단을 담당하고, 도구는 외부 정보 검색을 담당하며, 시스템 프롬프트는 행동 규칙을 정의한다.

### 2-4. 질문 분석 및 도구 호출

에이전트가 사용자 질문의 의도를 파악한 뒤, 시스템 프롬프트에 명시된 조합 패턴에 따라 2개 이상의 도구를 자율적으로 선택·호출한다. 호출 가능한 도구는 다음 3가지이다.

| 도구 | 데이터 소스 | 역할 |
|------|------------|------|
| `search_symptoms` | Elasticsearch (BM25) | 증상·질환 관련 의료 문서 검색 |
| `get_medication_info` | 식약처 "e약은요" API | 약물 효능·부작용·주의사항 조회 |
| `find_nearby_hospitals` | 건강보험심사평가원 API | 지역·진료과목 기반 병원 목록 조회 |

### 2-5. 답변 생성

에이전트가 도구 실행 결과를 종합하여 최종 답변을 생성한다. 이때 시스템 프롬프트의 제약에 따라, 도구 검색 결과에서 확인된 정보만을 근거로 답변하며 근거 없는 내용은 포함하지 않는다.

### 2-6. 스트리밍 응답

전체 과정은 SSE(Server-Sent Events) 방식으로 실시간 전달된다. 흐름은 다음과 같다.

```
① {"step": "model", "tool_calls": ["search_symptoms"]}         ← 도구 호출 시작
② {"step": "tools", "name": "search_symptoms", "content": ...}  ← 도구 실행 결과
③ {"step": "model", "tool_calls": ["get_medication_info"]}       ← 다음 도구 호출
④ {"step": "tools", "name": "get_medication_info", "content": ...}
⑤ {"step": "done", "content": "고혈압은 ...입니다", ...}         ← 최종 답변
```

클라이언트는 각 단계를 수신할 때마다 화면에 표시할 수 있으므로, 사용자는 에이전트가 현재 어떤 작업을 수행 중인지 실시간으로 확인할 수 있다.

## 3. 도구(Tool) 상세 구현

### 3-1. search_symptoms — Elasticsearch BM25 기반 의료 문서 검색

Elasticsearch에 연결하여 BM25 알고리즘으로 증상 및 질환 관련 의료 문서를 검색한다. BM25란 문서 검색에서 가장 널리 사용되는 랭킹 알고리즘으로, 단어의 출현 빈도(TF)와 전체 문서 내 희귀도(IDF)를 고려하여 관련도 점수를 산출한다.

검색 쿼리의 `operator`를 `"or"`로 설정하였으므로, "두통 어지러움"으로 검색하면 두 단어 중 하나라도 포함된 문서를 모두 검색하되, 두 단어가 모두 포함된 문서에 더 높은 점수를 부여한다. 상위 10건(`_TOP_K = 10`)을 반환하며, 각 결과에 출처(`source_spec`)와 작성 연도(`creation_year`)를 메타데이터로 포함하여 정보의 신뢰도를 판단할 수 있도록 하였다.

### 3-2. get_medication_info — 식약처 공공 API 연동

식품의약품안전처의 "e약은요" API를 호출하여 약물 정보를 조회한다. 약물 이름으로 검색하면 효능, 사용법, 주의사항(경고), 일반 주의사항, 상호작용, 부작용, 보관법의 7가지 항목을 반환한다. 정부 공인 데이터베이스에서 직접 가져오므로 정보의 신뢰성이 높다.

### 3-3. find_nearby_hospitals — 건강보험심사평가원 API 연동

건강보험심사평가원의 병원정보서비스 API를 호출하여 병원 목록을 조회한다. 이 도구에는 시도 코드, 병원 종별 코드, 진료과목 코드의 3가지 매핑 테이블이 내장되어 있다. 사용자가 "서울 내과"라고 입력하면 "서울"을 시도 코드 `110000`으로, "내과"를 진료과목 코드 `01`로 변환하여 API를 호출한다. 자연어를 API가 이해할 수 있는 코드로 변환하는 중간 레이어 역할을 수행하며, 지역명 매핑에 실패하면 병원명 검색으로 자동 전환(fallback)하는 예외 처리를 포함한다.

## 4. 시스템 프롬프트 설계

시스템 프롬프트는 에이전트의 행동 방식을 결정하는 핵심 요소로, 답변 품질의 대부분을 좌우한다. 설계 시 다음 세 가지 원칙을 적용하였다.

### 4-1. 도구 조합 필수화

단일 도구만 호출하면 답변이 불완전해진다. "고혈압에 대해 알려주세요"라는 질문에 질환 설명만 제공하고 관련 약물 정보를 생략하면 불충분한 답변이 된다. 따라서 질환/진단 질문에는 `search_symptoms + get_medication_info`를, 약물 질문에는 `get_medication_info + search_symptoms`를 반드시 함께 호출하도록 패턴을 명시하였다.

### 4-2. 근거 기반 답변 제한

LLM은 학습 데이터에서 "그럴듯한" 의료 정보를 생성할 수 있으나, 의료 도메인에서 이는 환자 안전을 위협할 수 있다. 이를 방지하기 위해 "검색 결과에서 확인되지 않은 구체적 수치, 통계, 치료법을 언급하지 마세요"라는 제약 조건을 명시하였다. 또한 검색 결과가 부족한 경우에는 "더 자세한 정보는 전문의와 상담하세요"로 안내하도록 유도하여, 모르는 것에 대해 모른다고 답하는 구조를 확보하였다.

### 4-3. 의료 면책 조항 포함

AI가 제공하는 정보는 진단이나 처방을 대체하지 않으며, 정확한 진단을 위해 반드시 전문의를 방문해야 한다는 점을 답변에 명시하도록 설계하였다.

## 5. 평가 데이터셋 구성

`opik_dataset.json`에 30명의 가상 환자 레코드를 구성하였다. 각 레코드에는 나이, 성별, 진단명, 검사 결과, 처방 정보가 포함되어 있으며, 감기, 고혈압, 당뇨병, 협심증, 외이도염 등 다양한 질환을 포함하여 에이전트의 범용성을 테스트할 수 있도록 설계하였다.

각 환자 레코드는 평가 시 다음과 같이 변환된다.

| 키 | 내용 | 용도 |
|----|------|------|
| input | 자연어 질문 (예: "35세 여 환자입니다. 감기 진단을 받았습니다...") | 에이전트에 전달할 질문 |
| reference | 기대 진단명 (예: "감기") | Contains 메트릭의 비교 대상 |
| context | 근거 정보 리스트 (예: 진단, 검사 결과, 처방) | Hallucination·ContextRecall 메트릭의 근거 |

---

## 6. 평가 메트릭 선택과 그 이유

의료 AI의 답변 품질을 판단하기 위해서는 먼저 "좋은 답변이란 무엇인가"를 정의해야 한다. 본 프로젝트에서는 Heuristic 메트릭 1개와 LLM-as-a-judge 메트릭 3개, 총 4개의 메트릭을 선택하였다.

### 6-1. Contains (diagnosis_mentioned) — 주제 정확성

에이전트 응답에 기대하는 진단명이 문자열로 포함되어 있는지를 확인하는 Heuristic 메트릭이다. 이는 가장 기본적인 검증 단계로, "고혈압에 대해 알려주세요"라는 질문에 대해 답변에 "고혈압"이라는 단어 자체가 없다면 근본적으로 잘못된 답변이기 때문이다. Heuristic 방식이므로 빠르고, 결과가 항상 동일하며(deterministic), LLM 호출 비용이 들지 않는다는 장점이 있다.

### 6-2. Hallucination (hallucination_check) — 허위 정보 생성 여부

에이전트가 제공된 컨텍스트(검사 결과, 처방 정보)에 근거하지 않은 허위 정보를 생성했는지를 LLM이 판단하는 메트릭이다. 의료 도메인에서 **가장 중요한 메트릭**에 해당한다.

예를 들어, 에이전트가 "이 약의 부작용으로 3.7%의 환자에서 간수치 상승이 보고되었습니다"라고 답변했으나 실제 도구 검색 결과에 해당 수치가 없는 경우, 이는 LLM이 학습 데이터에서 그럴듯하게 생성한 정보이다. 의료 정보에서 이러한 "그럴듯한 거짓"은 환자에게 직접적인 해를 끼칠 수 있으므로 반드시 측정해야 한다.

### 6-3. AnswerRelevance (answer_relevance) — 질문 의도 부합 여부

에이전트의 답변 전체가 질문의 의도와 의미적으로 관련이 있는지를 LLM이 판단하는 메트릭이다. Contains가 "단어 포함 여부"만을 확인하는 반면, AnswerRelevance는 답변 전체의 의미적 관련성을 평가한다.

"고혈압 약 부작용이 뭔가요?"라는 질문에 대해 답변의 대부분이 고혈압의 원인과 예방법에 관한 내용이라면, 진단명 "고혈압"은 포함되어 있으므로 Contains는 통과하지만 정작 질문한 "부작용"에 대한 답변은 부족한 셈이다. AnswerRelevance는 이러한 경우를 포착한다.

### 6-4. ContextRecall (context_recall) — 정보 활용도

제공된 컨텍스트(진단명, 검사 결과, 처방 정보)를 에이전트가 답변에 얼마나 충분히 반영했는지를 평가하는 메트릭이다. 에이전트가 도구를 호출하여 정보를 가져왔으나 정작 답변에서 해당 정보를 활용하지 않는 경우를 감지한다. 예컨대 context에 "처방: 해열제, 진해제"가 있으나 답변에서 약물 관련 언급이 전혀 없는 경우 ContextRecall이 낮게 산출된다.

  │ ContextRecall   │ 높을수록 좋음 │ 1.0       │ 0.0       │ 검사결과·처방 정보를 잘 활용        │
  └─────────────────┴───────────────┴───────────┴───────────┴─────────────────────────────────────┘

이 4가지를 모두 통과해야 "좋은 의료 상담 답변"으로 간주할 수 있으며, 각각 **정확성**, **신뢰성**, **관련성**, **정보 활용도**라는 서로 다른 품질 차원을 담당한다.

## 8. 평가 실행 구조

평가는 Opik의 5단계 프로세스를 따라 진행하였다.

### 8-1. 트레이싱 설정 (1단계)

`@track` 데코레이터를 3개 도구 함수와 에이전트 팩토리 함수에 적용하고, `OpikTracer`를 LangGraph 에이전트에 통합하여 전체 실행 흐름을 추적하도록 구성하였다.

### 8-2. 평가 태스크 정의 (2단계)

`evaluation_task` 함수에서 데이터셋 항목을 받아, 매 호출마다 독립적인 에이전트를 새로 생성한다. 독립 생성의 이유는, 이전 평가의 대화 기록이 다음 평가에 영향을 주면 공정한 결과를 얻을 수 없기 때문이다. 비동기 스트리밍으로 에이전트를 실행한 뒤, `ChatResponse`의 content를 추출하여 메트릭이 기대하는 형식으로 반환한다.

### 8-3. 데이터셋 생성 (3단계)

30개 환자 레코드를 Opik 데이터셋으로 변환하여 등록한다. 기존 데이터셋이 존재하면 중복 생성을 방지하는 로직이 포함되어 있다.

### 8-4. 메트릭 적용 및 실행 (4-5단계)

4개 메트릭을 `scoring_metrics`로 등록하고, `scoring_key_mapping`으로 데이터셋 키와 메트릭 입력을 매핑한 뒤, `opik.evaluation.evaluate()`를 호출하여 전체 평가를 실행한다. `task_threads=2`로 설정하여 2건의 평가를 동시에 병렬 처리하되, API rate limit 초과를 방지하기 위해 과도한 병렬화는 지양하였다.

### 8-5. 전체 평가 흐름 요약

```
opik_dataset.json (30개 환자 레코드)
        ↓
    create_dataset()  →  Opik 데이터셋으로 변환
        ↓
    evaluation_task()  →  각 항목마다 독립 에이전트 생성 → 질문 → 답변 수집
        ↓
    evaluate()  →  4개 메트릭으로 채점
        ↓
    Opik 대시보드에서 결과 확인 및 분석
```

## 9. 종합 정리

본 시스템은 LLM이 자율적으로 3가지 외부 도구를 조합 호출하여 의료 답변을 생성하며, Opik을 활용하여 정확성(Contains), 신뢰성(Hallucination), 관련성(AnswerRelevance), 정보 활용도(ContextRecall)의 4가지 품질 차원을 정량적으로 측정한다. 평가 결과를 기반으로 프롬프트 제약 강화, 도구 조합 필수화, 트레이싱 기반 병목 분석을 반복 수행함으로써 에이전트의 답변 품질을 지속적으로 개선하는 구조를 갖추고 있다.
